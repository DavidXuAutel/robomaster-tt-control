# Tello 无人机 · 避障能力与数据交接

> 本文与同目录 `2026-07-21-tello-avoidance-and-data-handover.html` 内容一致(纯文本版,便于检索/注入)。

---

## 0. 本轮测试目标(一次测试 = 1 个验证 + 2 类数据)

本轮 Tello 避障测试可概括为三个目标:

**目标一 · 验证避障能力边界** —— 验证「单目相机 + Depth Anything V2 Small + 反应式控制律」在真机上的能力与失效边界:接近减速 / 拐弯绕障(TURN)/ 被围悬停(BLOCKED)三种行为成立、失败降级链可触发、盲区(玻璃/白墙/细线)确认、参数整定与重复性(≥5 次一致)。

**目标二 · 产出两类数据:**

| | 2a 仿真轨迹校正数据 | 2b 自由避障飞行视频数据 |
|---|---|---|
| 来自哪种飞行 | 校正飞行(同向 pad、直线、**不转向**) | 避障飞行(自主避障、含转向,pad 可选) |
| 产出 | 真机位姿 + 动作(`frames.csv`) | `video.mp4` + `frames.csv`(动作+本体状态) |
| 给谁用 | `sim_match_report.py` 出可信度报告 | 喂世界模型做具身特定后训练 |
| 作用 | 证明"**仿真是真机的可信替身**"(对准/校正) | 让 WAM 学**无人机的因果动力学** |

**三者关系:** 目标一与 2b 在同一批避障飞行里同时完成(飞时开 `--record` 即采数据);2a 是单独的受控校正飞行(避障要转向会破坏 Mission Pad 轨迹拼接,故校正须"不转向 + 同向 pad"单独飞)。

## 1. 定位:无人机在数据飞轮里是什么角色

Tello 无人机是**同一套"数据飞轮 + 世界模型(WAM)"方法论里、更廉价更安全的第二个具身载体**。它做两件事:

1. **用视觉避障当安全网** —— 让飞机能不炸机地自主飞;
2. **每次飞行同步产出训练级数据**(视频 + 轨迹) —— 供仿真校正与世界模型微调。

一句话:**避障是"采数据的司机",WAM 才是要培养的大脑。** 与机械臂(Franka+GELLO)那条腿同构——给同一个世界模型多贡献一种异构具身的数据与一个安全试验台。

---

## 2. 避障能力边界

本方案的"感知智能"只来自**一个**神经网络:**Depth Anything V2 Small**(单目深度估计,约 24.8M 参数,跑在远端 4090 GPU 微服务)。其余是写死的**反应式规则**:把深度切左/中/右三区、比阈值、锁向绕行。

关键指标:
- **单目相对深度** —— 看得出谁近谁远,但**给不出"米"**(度量尺度需 Mission Pad 位姿另行提供);
- **~9 fps** —— 端到端感知帧率(含 WiFi + 推理 ~106ms);
- **3 区** —— 决策粒度:左 / 中 / 右;
- **反应式** —— 无地图、无目标、无记忆。

### ✅ 能做到

| 能力 | 说明 |
|---|---|
| 躲大块障碍 | 躲开正前方**有纹理、不透明**的大块障碍(墙、纸箱、家具、门框):近了明显"变红"→ 触发避让 |
| 接近减速 | 越近前进杆量越小(线性刹车),不硬冲 |
| 朝空旷侧拐弯 | 向更空的一侧锁向偏航绕行(方向锁定不横跳) |
| 被围悬停 | 左中右都近 → 原地悬停(BLOCKED),不乱撞 |

### ⚠️ 做不到 / 会失效

| 场景 | 原因 |
|---|---|
| 测绝对距离 | 相对深度,无米制尺度(尺度靠 Mission Pad) |
| 玻璃/白墙/反光/细线 | 深度模型**看穿或看错** → 近度不升高,可能直接撞。**首要安全盲区** |
| 飞向目标点 / 认路 | 反应式无目标无地图,只会"别撞上、往开阔处走",**不会去指定位置、不会回家**。这是 WAM 的职责 |
| 穿窄缝 | 三区粒度不足以精细过缝 |
| 预测动态障碍 / 识别物体 | 单帧无时序;只出深度、无语义(分不清人/门/门帘) |

> **安全底线:** 因透明/反光/细线盲区真实存在,避障**永不作为唯一安全保障**——人工键盘接管 + 看门狗(挂载超时/感知失联自动悬停)是第一防线。挡板须"大、不透明、有纹理"。

> **一句话边界:** 它能"安全地别撞上有纹理的大块障碍",做不了"量距离 / 认路 / 去目标 / 穿窄缝 / 躲玻璃"。**导航与精细操作是 WAM 的活,避障只负责安全网。**

## 3. 飞行产生的数据 & 两大作用

每次飞行产出一个 **episode**:**视频(MP4)+ 轨迹数据(frames.csv)+ 深度网格**。两大用途:

### 作用一:仿真校正(对准 / 可信度锚定)

用真机记录的**动作**重放仿真运动学,量化"仿真轨迹 vs 真机轨迹"的偏差,产出**可信度报告**(逐轴 MAE/RMSE、航向误差、终点误差、路径长度比 + 真机实线 vs 仿真虚线叠图)。这是"仿真是真机可信替身"的量化证据,是后续仿真增广、想象 rollout 的地基。

### 作用二:世界模型微调(具身特定后训练)

飞行的 **(视频, 动作, 本体状态)** 序列 = 世界模型学习**无人机因果动力学**所需的(观测, 动作, 转移)三元组。教模型"给这个动作,画面就这样流动",把通用 WAM 适配到无人机本体。

| 数据用途 | 避障自主飞的数据是否合适 | 说明 |
|---|---|---|
| 训世界模型(空间/动力学) | ✅ 合适 | 要的是多样(观测,动作,转移)覆盖,动作谁产的不重要 |
| 仿真校正 | ✅ 合适 | 真机轨迹是校准仿真的真值锚 |
| 训动作策略/模仿学习 | ✕ 价值低 | 需人遥操作的有目的演示(本方案暂不涉及) |

> **两个要点:**
> 1. **视频不是"只能回看"** —— 对世界模型它是核心训练输入(视频经 VAE 编码后预测未来)。
> 2. **尺度锚(Mission Pad)只在"校正"时需要**;采世界模型数据可无 pad(要的是动作 + 画面变化的对齐)。单目深度本身无米制尺度。

## 4. 数据格式

一次飞行 = 一个 episode 目录。**视频是主观测,frames.csv 是逐帧索引兼轨迹,深度为辅助**。对齐 DROID/LeRobot"MP4 + 表格"的通行做法。

```
logs/episodes/ep_<YYYYMMDD_HHMMSS>/
├── meta.json      回合级元数据(含 video 信息 / 尺度锚 / 动作来源 / 结局)
├── video.mp4      H.264(CRF18);第 N 帧 <-> frames.csv 第 N 行(严格 1:1)
├── frames.csv     逐帧索引 + 轨迹(核心)
└── depth/000001.npy   96×128 float16 近度网格(按 ts 去重,不重复落盘)
```

### 4.1 `frames.csv` 关键字段(一帧一行,`t_mono_ms` 为对齐基准)

| 字段 | 含义 |
|---|---|
| `t_mono_ms` / `video_frame` | 单调时钟毫秒(真实时刻,对齐基准) / 对应视频 0 基帧号 |
| `pos_x/y/z_cm`, `yaw_deg` | 真机位姿真值(相对 Mission Pad,**尺度锚**);`pad_id<0` 时此行位姿不可比 |
| `act_roll/pitch/throttle/yaw` | **动作**:RC 杆量(归一化 -100~100,非米/速度,换算见下) |
| `vgx/vgy/vgz`, `pitch_deg/roll_deg`, `height_cm`, `bat_pct` | 本体状态:速度 / 姿态 / 高度 / 电量 |
| `depth_path`, `has_depth`, `depth_rtt_ms`, `depth_age_ms` | 对齐的深度网格 / 是否有效 / 推理耗时 / 深度滞后(配对深度相对本帧的滞后毫秒,筛查不同步坏帧用) |
| `ctrl_state`, `near_left/mid/right` | 决策状态(CRUISE/TURN/BLOCKED/MANUAL) / 左中右近度 |

### 4.2 动作语义(训练必读)

`act_*` 是归一化杆量,转真实速度用同一套运动学常数:

```
v_forward = (act_pitch  / 100) · VMAX      # VMAX   = 0.6 m/s
v_right   = (act_roll   / 100) · VMAX
v_up      = (act_throttle/ 100) · VZMAX     # VZMAX  = 0.5 m/s
yaw_rate  = (act_yaw    / 100) · YAWRATE    # YAWRATE= 70 °/s
```

### 4.3 `meta.json`(回合级,实现产出)

```json
{
  "episode_id": "ep_20260721_153000",
  "drone": "192.168.10.1", "sim": false,
  "action_source": "avoidance|manual|mixed",
  "camera": {"model": "tello-front-mono", "res": "960x720"},
  "depth": {"model": "DepthAnythingV2-Small", "semantic": "nearness(relative)"},
  "scale_anchor": "mission_pad",
  "video": {"file": "video.mp4", "codec": "h264", "crf": 18,
            "nominal_fps": 10, "frames": 0, "width": 960, "height": 720},
  "n_frames": 0, "n_depth_frames": 0, "depth_grid": [96, 128],
  "frames_manual": 0, "frames_auto": 0,
  "battery_start_pct": 92, "battery_end_pct": 76,
  "outcome": "completed|aborted|land_failed", "abort_reason": null,
  "t_start_ms": 0, "t_end_ms": 0, "notes": {}
}
```

### 4.4 校正输出(仿真可信度报告)

由 `sim_match_report.py` 产出,与 episode 同目录:

```
match_report.json   逐轴 MAE/RMSE(m) · 航向 MAE(deg) · 终点误差(m) · 路径长度比
match_report.png    真机实线 vs 仿真(动作重放)虚线 俯视叠图
```

### 4.5 同步保证

- 单一单调时钟 `t_mono_ms` 为唯一对齐基准;
- 视频帧与 CSV 行**严格 1:1**(帧↔动作对齐靠行号,不靠 MP4 名义帧率);
- 深度按 `ts` 去重 + 记 `depth_age_ms`,如实暴露 RGB↔深度滞后,便于筛查坏帧。

## 5. 两类飞行:采集飞行 vs 校正飞行

因为避障会偏航转向,而 Mission Pad 轨迹拼接假设"不旋转",两类目的**分开飞**:

| 类型 | 目的 | Mission Pad | 动作 |
|---|---|---|---|
| 校正飞行(少量) | 标定仿真可信度 | 沿直线铺**同向** pad、间距 ≤0.5m、机头对准 | 前进/后退等**不转向**的简单动作 |
| 避障/采集飞行(大量) | 喂世界模型微调 | 可有可无(有则记) | 避障自主飞(含转向) |

> 注:避障用前视相机、pad 用下视相机,两者独立可同时开;但避障转向会破坏 pad 拼接,故精确绝对轨迹只在"校正飞行"里取。

## 6. 待大众确认的对接点

1. **深度分辨率**:世界模型是自带深度分支、只吃 RGB(则现状 96×128 相对近度即可),还是要高分辨率/原始逆深度(需给感知服务加 raw 输出模式)?
2. **数据 schema**:是否要严格对齐大众侧已有的 DROID-1K Spatial / Scout-GELLO 字段命名与目录?若有定义,可直接对齐 `frames.csv`,省返工。
3. **WAM 对无人机的接口**:期望输入(RGB? RGB-D? 多视角?)与输出(速度场? 离散动作? 频率?)。无人机侧已预留 `ExternalModelPolicy` 适配位:把动作增量映射为 RC 四通道杆量,仲裁优先级 **键盘 > WAM > 避障 > 悬停**,接口一确认即可接入。
